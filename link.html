Link

The Link component is used for navigation within body text (inline) or as a standalone call-to-action. It supports three sizes, two surface contexts, and a full set of interactive states.

Interactive states:
  • Default — resting state; color token applied from surface context
  • Hover — link darkens to guide the user's attention
  • Pressed — deepest shade confirms the click or tap
  • Focus — 4px ring shown on keyboard navigation; meets WCAG 2.4.11
  • Disabled — muted grey; pointer-events: none prevents interaction

Properties

Property Type Default Description
href string — Destination URL the link navigates to.
style "standalone" | "inline" "standalone" Visual treatment. Standalone has a trailing arrow icon and no underline; inline is underlined with no icon.
size "sm" | "md" | "lg" "md" Controls font size and icon dimensions. Large uses Lato Bold (700).
surface "surface-01" | "card" "surface-01" Background context used to select the correct colour token set. Card uses a distinct dark-mode value for contrast.
disabled boolean false Prevents interaction and applies the disabled colour token. Set aria-disabled="true" and remove from tab order with tabindex="-1".

Usage

Use standalone for navigation actions
  • Place standalone links after a block of content as a clear "read more" or "view all" CTA
  • Use in card footers, empty states, and sidebars where the link stands alone
  • Choose Large (Bold) for prominent sections that need extra visual weight
Use inline within body text sentences
  • Wrap specific words or short phrases — not entire sentences — with an inline link
  • Match the link size to the surrounding paragraph size for visual consistency
  • Provide descriptive link text that makes sense out of context
Don't use links as buttons for primary actions
  • If the action submits data, triggers a modal, or changes application state, use a Button instead
  • Avoid replacing Button components with styled links purely for aesthetic reasons
  • Links imply navigation; buttons imply action — preserve this semantic distinction
Don't mix sizes in the same sentence
  • An inline link must match the font size of the surrounding paragraph text
  • Avoid placing a Large link inside a paragraph written in 14px body copy
  • Keep standalone link sizes consistent within the same section or card

Accessibility

Keyboard Navigation

  • Tab moves focus to the next link in the natural document order. Ensure disabled links are removed from the tab sequence with tabindex="-1".
  • Enter activates the focused link and follows the href destination.
  • The focus ring (box-shadow: 0 0 0 4px rgba(90,191,224,0.35)) is always visible on keyboard focus to satisfy WCAG 2.4.11 Focus Appearance.

Screen Reader

  • Use descriptive link text that makes sense when read out of context. Avoid generic phrases like "click here" or "read more" without additional context.
  • When a link opens in a new tab, add target="_blank" together with aria-label="... (opens in a new tab)" or a visually hidden notice so screen reader users are informed.
  • Disabled links should carry aria-disabled="true" so assistive technology announces their unavailability rather than silently skipping them.
  • Standalone links with icon-only trailing arrows must include visible text — the icon is decorative and should carry aria-hidden="true" if rendered as SVG outside of Lucide's auto-creation.

Colour & Contrast

  • All link colour tokens meet the WCAG AA 4.5:1 contrast ratio against their intended surface backgrounds in both light and dark modes.
  • Inline links supplement colour with an underline so they are distinguishable from surrounding text for users with colour vision deficiencies.
  • The disabled colour (#6B7280) is intentionally below 4.5:1 to communicate unavailability — this is an accepted pattern per WCAG 1.4.3 exception for disabled controls.
