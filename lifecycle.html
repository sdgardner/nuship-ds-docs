Process

From Figma to Claude Design — the component lifecycle.

Every nuShip component starts in Figma with tokens already applied and ends as a published, dev-reviewed asset in Claude Design. Here's the five-step path it takes, and what to do at each stop.

Five steps · One source of truth

The flow

Lifecycle at a glance

Designers start with existing tokens. If a new token is needed, it's added in two places before anything moves downstream. Dev review gates every push to GitHub, and Claude Design is the final stop.

Step by step

What happens at each stop

One card per step. Each lists the tool you're in, who's involved, and what counts as "done" before you move on.

01

Apply existing tokens

Build the component in Figma using only published variables. Every fill, stroke, radius, and spacing reads from a token — no raw hex, no detached values.

Figma Designer
02

Create new tokens if needed

If the design needs a value that doesn't exist yet, add it twice — once to the Variables chart in Figma, once to Token Studio. Name it by role, not by value. Skip the step if existing tokens cover the design.

Figma Variables Token Studio Designer
03

Check with developers

Before anything ships to GitHub, walk the new tokens through with engineering. Names, semantics, naming-collision risks, and dark-mode pairings all get reviewed here. This is a hard gate.

Slack / sync Designer + Engineering
04

Push tokens to GitHub

Once devs are aligned, push the Token Studio export to the repo. Open a PR with the token diff, dark-mode pair, and a screenshot of the component using it.

GitHub Designer
05

Update Claude Design

After the PR merges, publish the updated component to Claude Design so the docs site reflects the new state. The component is now part of the system.
