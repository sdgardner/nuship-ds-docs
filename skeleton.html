Skeleton

Skeletons are placeholder shapes that hold the position of content while it loads, with a subtle shimmer animation that signals progress without distracting the user. Two variants are available — Text for paragraph or label placeholders and Area for rectangular blocks like cards, images, or media regions.

2Shimmer highlight (sweeps left → right)

Variants

Text — use when:
  • Reserving space for a paragraph, label, or list of strings
  • Each line uses 2px radius and a 16px gap below
  • Vary line widths (50%, 70%, 100%) so the placeholder reads as natural prose
Area — use when:
  • Reserving space for a card, image, media block, or icon
  • Default 6px radius; override to 50% for circular avatars
  • Set explicit width and height to match the real content's footprint
Composed — use when:
  • Mirroring a real layout (e.g. avatar + name + body) so the load-in is seamless
  • Combine Area and Text primitives — keep widths and gaps consistent with the final UI
  • The shimmer animations stay in sync because they share a single keyframe timing

Usage Guidelines

Mirror the real layout
  • Match the size and position of the content that will replace the skeleton
  • Use distinct shapes for text lines, avatars, images, and buttons
  • Preserve container padding so the layout doesn't jump on load
Use for initial load
  • Show skeletons during the first paint when the page structure is known
  • Replace skeletons in place as each section's data resolves
Keep animation subtle
  • Use a gentle shimmer or pulse — fast enough to feel alive, slow enough to feel calm
  • Respect prefers-reduced-motion by disabling animation for users who opt out
Communicate state
  • Mark the skeleton region with aria-busy="true" so assistive tech can announce loading
  • Provide a screen-reader-only "Loading…" label for skeleton groups
Use for fast loads
  • Don't show a skeleton when the data resolves in under ~300ms — it just flashes
  • Don't add artificial delay so the skeleton "looks impressive"
Mismatch real layout
  • Don't use a generic skeleton when the loaded layout differs significantly — it causes layout shift
  • Don't omit elements that will appear on load (e.g. the avatar) — the layout will jump
Replace error states
  • Don't show a skeleton indefinitely if the request fails — switch to an inline error or retry state
  • Set a reasonable timeout (e.g. 10s) before transitioning to an error view
Distract with motion
  • Don't use aggressive or contrasting shimmer that competes with surrounding content
  • Don't animate multiple skeletons out of sync — keep the motion uniform across the page

Accessibility

  • ARIA: Wrap the skeleton group in role="status" with aria-busy="true" and aria-live="polite" so screen readers announce the loading state without dumping the placeholder shapes.
  • Hide decorations from AT: Each individual shape should have aria-hidden="true". The visible "Loading…" announcement comes from the wrapping live region, not the skeleton bars themselves.
  • Reduced motion: Honor prefers-reduced-motion by disabling the shimmer keyframes — the static gray track is still a valid loading indicator.
  • Don't show forever: If a request takes longer than ~10s, replace the skeleton with a fallback message so users aren't stuck staring at an animation.
