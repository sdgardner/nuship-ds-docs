Overlays

Modal

The Modal component displays focused content in a layer above the main page, requiring user interaction before returning to the primary workflow. Modals temporarily block interaction with the underlying page until the user completes the required action or dismisses the dialog.

Info (Default) — Use when:
  • The action requires user confirmation but carries no destructive consequence
  • Presenting a neutral prompt or decision point
  • Default choice when no specific semantic type applies
Visual changes:
  • Icon uses color/text/color/icon/info (blue)
  • Confirm button uses color/action/secondary/default (blue)
Example contexts:
  • Confirming a shipment address before submission
  • Prompting the user to save changes before navigating away
  • Verifying an action that affects a booking

Usage

Consequential or Irreversible Actions
  • When the user is about to do something that cannot be undone
  • Deleting a shipment, canceling a booking, removing saved information
  • When the cost of an accidental action is high
Explicit User Confirmation
  • When you need a deliberate, conscious decision from the user
  • Actions that affect other users or external systems
  • When passive dismissal (like navigating away) is not sufficient
Brief Input Collection
  • When a small amount of information is needed before proceeding
  • 1–2 fields maximum — address confirmation, contact details
  • When the input is directly tied to the action being confirmed
Status Updates and Notifications
  • Confirming something completed successfully — use a toast
  • Displaying errors that don't require a decision — use inline validation
  • Non-critical system messages
Large Amounts of Content
  • Information that requires scrolling to read in full
  • Multi-step flows or wizards — use a dedicated screen
  • Content that benefits from its own navigation context
Low-Stakes Actions
  • Actions that are easily reversible
  • Routine tasks that don't require confirmation
  • When interrupting the user's flow isn't justified by the action's risk
Stacked or Chained Modals
  • Triggering a modal from within another modal
  • Using a modal as a navigation destination
  • Sequences that could be handled in a single screen

Accessibility

Keyboard Shortcut

  • Tab — move focus between interactive elements inside the modal (close button, Cancel, Confirm).
  • Shift + Tab — move focus backwards through modal elements.
  • Enter / Space — activate the focused button.
  • Escape — dismiss the modal (equivalent to Cancel).

Screen Readers

  • Modal container uses role="dialog" with aria-modal="true".
  • aria-labelledby points to the modal title element.
  • aria-describedby points to the description element.
  • Screen reader announces modal title when it opens.
  • Backdrop/scrim marked with aria-hidden="true" to prevent background content from being read.

Focus Management

  • Focus moves into the modal immediately on open — do not leave focus on the triggering element.
  • Focus is trapped inside the modal while it is open — Tab should not reach background content.
  • On close, focus returns to the element that triggered the modal.

Visual

  • Confirm button color must meet WCAG AA contrast (4.5:1) across all three semantic types.
  • Icon alone does not convey type — title and button color reinforce semantic meaning.
  • Close button (×) must have a visible focus ring.
