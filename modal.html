Modal

A focused dialog that interrupts the current workflow to request user confirmation, present critical information, or complete a short task. Modals support three intents and two screen sizes.

Properties

screenSize "desktop" | "mobile" "desktop" Desktop shows 400px with horizontal buttons; mobile shows 287px with stacked full-width buttons.
showIcon boolean true Toggles the leading 24×24 intent icon in the header.
showClose boolean true Toggles the X close button in the header.
title string — Modal heading text.
description string — Body copy describing the action or context.
confirmLabel string "Confirm" Label for the primary action button.

Accessibility

  • Keyboard navigation: When the modal opens, focus is moved to the first focusable element inside the dialog. Tab cycles forward through interactive elements; Shift+Tab cycles backward. Pressing Enter or Space activates the focused button. Pressing Esc dismisses the modal and returns focus to the triggering element.
  • Focus trap: Keyboard focus must be contained within the modal while it is open. Focus must not reach elements in the background page. Implement a focus trap loop so that tabbing past the last element wraps back to the first, and shift-tabbing before the first wraps to the last.
  • ARIA roles: The modal container must have role="dialog" and aria-modal="true" to inform assistive technologies that the dialog is modal. Use aria-labelledby pointing to the title element and aria-describedby pointing to the body text so screen readers announce context immediately on focus entry.
  • Background inertness: When the modal is open, background content should be made inert (using the inert attribute or equivalent) to prevent screen reader and keyboard access to the underlying page until the modal is closed.
