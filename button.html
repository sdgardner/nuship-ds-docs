Actions

Button

A button is an interactive control that initiates an action when a user clicks or taps it. Buttons are used for clear, immediate operations such as submitting a form, saving changes, or navigating to another view. Destructive buttons remove or permanently change data. Their visual weight and required confirmation should match the level of risk.

Variants

Use when:
  • The primary button is the main call to action on a page. Use it for the key task you want the user to complete.
Example content:
  • One per context: Use a single primary button per page, modal, or independent card
  • Placement: Align it with the main task (e.g., "Continue", "Save", "Book now")
  • Priority: Reserve it for high-value or irreversible actions and avoid overusing it alongside other strong visuals
Use when:
  • Secondary buttons support the main flow and are used for confirm or progress actions in modals, drawers, and step flows. They're also used for in-page actions such as "Edit", "Select", "View", or "Apply filters".
Example content:
  • Contextual primary: In contained surfaces (modals, drawers, wizards), secondary buttons can act as the main confirm action when a global primary already exists
  • In-page actions: Use for actions that stay on the same page (e.g., "Edit", "Select", "View", "Apply") without being the main page CTA
  • Pairing: When paired with a primary button, the secondary carries the supporting action (e.g., "Save draft", "Apply filters")
  • With ghost: If paired with a ghost button, the secondary should confirm or apply; the ghost should cancel or dismiss
Use when:
  • Tertiary buttons are low-emphasis actions used for small or repeated tasks that should not compete with the main page actions.
Example content:
  • Usage: Use for lightweight utilities such as "View details", "Add note", or "Manage"
  • Frequency: They can appear multiple times on a page without overwhelming the primary action
  • Priority: Do not use tertiary buttons for primary flows or high-impact decisions
Use when:
  • Ghost buttons are low-emphasis actions used for small or repeated tasks that should not compete with the main page actions.
Example content:
  • Usage: Use for lightweight utilities such as "View details", "Add note", or "Manage"
  • Frequency: They can appear multiple times on a page without overwhelming the primary action
  • Priority: Do not use ghost buttons for primary flows or high-impact decisions
High-impact deletion — use when:
  • Loss is significant or disruptive (e.g., delete an account or payment profile)
  • Require strong confirmation (typing a name, checking a box) and clearly describe the outcome
  • After a destructive action, return the user to the parent view, animate removal or show clear success feedback, and restore state when possible if an error occurs
Low-impact deletion — use when:
  • Destructive tertiary buttons are low-emphasis actions used for small or repeated deletions that should not compete with the main page actions
Example content:
  • Usage: Use for lightweight removals such as "Remove tag", "Discard draft", or "Clear filter"
  • Frequency: They can appear multiple times on a page without overwhelming the primary action
  • Priority: Do not use destructive tertiary buttons for high-impact or irreversible deletions
Moderate-impact deletion — use when:
  • Deletion is irreversible or affects multiple items (e.g., delete a listing or shipment/quote)
  • Require confirmation and clearly state what will happen
  • Recommended style: Secondary Destructive

Usage Guidelines

Size
  • Use Medium (48px) as the default for most UI: forms, modals, in-page actions. It aligns with input heights and keeps a consistent rhythm. When pairing with a button use medium since it matches the same hight.
  • Use Large (56px) for primary CTAs in hero sections, mobile-first flows, and critical modals where the action needs strong emphasis.
  • Use Small (32px) in dense layouts: tables, toolbars, side panels, and compact cards where space is limited.
  • Keep one size per button group (e.g. same size for "Cancel" and "Save" in a modal).
Type and hierarchy
  • Use one primary button per context (page, modal, or card).
  • Use secondary for in-page actions (Edit, Select, View, Apply) and for confirm/cancel in modals.
  • Use ghost for low-emphasis or dismissive actions (Cancel, Back, Manage).
Labels and layout
  • Use short, action-oriented labels (e.g. "Save", "Cancel", "Add item").
  • Keep icon + label to a single icon (left or right); use the same side consistently in a flow.
  • Keep button width fixed when switching to loading state to avoid layout shift.
Loading and disabled
  • Use disabled background for loading on filled buttons; use disabled text only (no fill) for ghost loading.
  • Place spinner left, loading text right (e.g. "Saving…"); use spinner only for icon-only buttons.
  • Use disabled state when the action isn't available; pair with validation or messaging when it's not obvious why.
Size
  • Don't use Small for critical or primary actions; reserve it for secondary/utility actions in tight layouts.
  • Don't mix button sizes in the same row or group.
  • Don't use Large for every button on a page; use it only where you need maximum emphasis.
Type and hierarchy
  • Don't use two primary buttons in the same view; it dilutes the main action.
  • Don't use ghost for primary or high-impact actions.
  • Don't use primary styling for destructive actions; use the destructive variants (primary/secondary/ghost destructive).
Labels and layout
  • Don't truncate button labels; use shorter copy or allow the button to grow within reason.
  • Don't use multiple icons in one button (except loading, where the spinner replaces the icon).
  • Don't rely on icon-only buttons without an accessible name (e.g. aria-label).
Loading and disabled
  • Don't change button width on transition to the loading state.
  • Don't show loading without a label when the spinner is the only feedback users get.
  • Don't leave disabled buttons unexplained — pair with inline validation or tooltip when needed.

Accessibility

  • Keyboard navigation: Buttons are reached via Tab and activated with Enter or Space. Shift + Tab moves focus backwards. If a button opens a dialog or menu, Escape dismisses it and returns focus to the trigger.
  • Icon-only buttons: Provide an accessible name via aria-label (e.g. aria-label="Close") since the icon alone is not announced by screen readers. Pair the icon with a tooltip on hover/focus so sighted users learn the label too.
  • Disabled state: Use disabled when the action is not available. Disabled buttons should not receive focus and should not be activated by keyboard. If the user needs to understand why a button is disabled, surface that reason in nearby helper text or a tooltip — don't rely on the visual state alone.
  • Focus indicator: The 2px blue ring + 4px glow meets WCAG 2.4.7 (Focus Visible) at 3:1 contrast against the page background. Never suppress focus rings on keyboard navigation, and verify visibility in both light and dark themes.
