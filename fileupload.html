File Upload

The Drop Zone is a file upload component that allows users to upload files by dragging and dropping them onto the target area or by clicking to browse their file system. It supports multiple files and provides per-file status feedback through the File Row sub-component.

Uploading — Use when:
  • Files have been selected and upload is actively in progress
  • One or more files may be queued, uploading, completed, or failed mid-batch
Example contexts:
  • Batch photo upload for a vehicle transport listing
  • Document submission during carrier onboarding
Success — Use when:
  • All files have finished uploading successfully
  • The user may need to review, add more, or remove files
Example contexts:
  • Confirming all supporting documents have been attached before submitting a claim
  • Photo upload complete before publishing a listing
Error — Use when:
  • One or more files failed to upload (size exceeded, unsupported type, network failure)
  • The batch completed but with partial failures
Example contexts:
  • User attempts to upload an oversized image
  • Network interruption causes mid-batch failure on one or more files

Usage

Multi-file selection with progress feedback
  • When users need to upload one or more files and benefit from seeing upload status per file
  • Batch uploads where individual file success/failure needs to be communicated
  • When file size and type constraints should be communicated upfront
Drag-and-drop primary workflows
  • Contexts where users are likely to have files readily accessible (desktop, file manager)
  • When the upload action is a primary task on the screen, not a secondary one
  • Document submission, photo uploads, attachment flows
Situations requiring upload validation
  • When file type or size restrictions need to be enforced with inline feedback
  • When partial failure (some files success, some fail) is a likely outcome and needs clear per-file status
  • When users may need to retry failed uploads without restarting the entire flow
Single, simple file replacement
  • Replacing a profile photo or avatar — use a targeted upload button instead
  • When the interaction is one-tap and doesn't benefit from drag-and-drop affordance
  • Contexts where a compact inline input fits better than a full drop zone
Mobile-primary experiences
  • Drag-and-drop has no affordance on touch devices — use a button-triggered file picker instead
  • The drop zone's large fixed height (260px) is not optimized for small viewports
  • When the primary user is on a carrier mobile device in the field
Mandatory blocking flows
  • Don't gate all page progression on file upload unless it's truly required
  • Avoid using the drop zone in a modal — combine with a dedicated screen instead
  • When the file is optional, consider a less prominent inline attachment pattern
Uploading files silently in the background
  • This component is designed to surface progress explicitly — don't use it if uploads should happen invisibly
  • If the user doesn't need to track individual file status, a simpler upload trigger is more appropriate
  • Avoid showing the Uploading state if the progress resolves too quickly to be meaningful

Accessibility

Keyboard Shortcut

  • The drop zone must be reachable and activatable via keyboard (Tab to focus, Enter / Space to trigger the file browser).
  • "Browse Files," "Add More," and "Retry Failed" buttons must all be in the natural tab order.
  • Each file row's action icon (cancel, delete) must be individually focusable and operable by keyboard.
  • Focus should not be trapped inside the component during upload.

Screen Readers

  • The drop zone container should have role="button" or be a native <button> or <input type="file"> with a visible label.
  • Drag-and-drop affordance must not be the only way to upload — the Browse Files button is the accessible fallback.
  • Upload progress should be announced dynamically — use aria-live="polite" on the status region so "Uploading…", "Complete", and error messages are read aloud without requiring focus.
  • The summary message ("3 files uploaded successfully" / "2 of 4 files failed") should also be in a live region.
  • File row action icons need descriptive aria-label values — e.g., "Cancel upload for photo_001.jpg" and "Delete photo_001.jpg".

Focus Management

  • When a file is removed or upload completes, focus should move logically — not drop to the top of the page.
  • After "Retry Failed" is triggered, focus should return to the file list region.
  • If the drop zone transitions from empty to file list view, screen readers should be informed of the context change.

Visual

  • Confirm button color must meet WCAG AA contrast (4.5:1) across all three semantic types.
  • Icon alone does not convey type — title and button color reinforce semantic meaning.
  • Close button (×) must have a visible focus ring.
