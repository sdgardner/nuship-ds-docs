Text Input

The Text Input is a single-line field for short freeform content such as names, addresses, and codes. It supports a label, optional leading or trailing icon, helper text, and multiple interaction states including error and read-only.

Usage

Do
  • Use a concise, descriptive label that clearly identifies the expected input.
  • Use placeholder text as an example, not a substitute for the label.
  • Use the leading icon to reinforce the input's purpose (e.g., a search icon for search, a map pin for location).
  • Show a helper text below the field to explain format requirements upfront.
Don't
  • Don't rely solely on placeholder text to communicate what the field requires — it disappears when typing begins.
  • Don't use disabled for fields that might become relevant; prefer read-only so screen readers can still access the value.
  • Don't show an error until after the user has had a chance to interact with the field (validate on blur or submit).
  • Don't use two icons simultaneously (leading and trailing) as this clutters the field.

Accessibility

Keyboard support

The field is fully reachable via Tab. Users can type directly when focused. Escape should clear or blur the field in most implementations.

Screen reader

Pair the visible label with for/id attributes on the underlying <input>. Use aria-describedby to associate helper and error messages.

Error announcement

When the error state activates, ensure the error message is surfaced via aria-live="polite" or role="alert" so screen reader users are notified immediately.

Icons

Decorative icons should have aria-hidden="true". If an icon conveys meaning (e.g., a clear button), provide an accessible label via aria-label.

Contrast

Both placeholder and value text meet WCAG AA contrast requirements against the field background in light and dark modes.
