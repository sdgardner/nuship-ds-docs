Slider

The Slider component allows users to select a numeric value from a continuous range by dragging a thumb along a track. Sliders are ideal for adjusting settings, filtering by price ranges, or selecting values where the relative magnitude matters more than the exact number.

States
  • Default — slider at rest, value badge visible above thumb
  • Hover — cursor over the thumb, shadow deepens
  • Focus — thumb receives keyboard focus, teal glow ring appears
  • Active — user is dragging, thumb grows slightly
  • Disabled — slider is not interactive, all elements muted

Usage Guidelines

Continuous Range Selection
  • Selecting approximate values from a range where precision isn't critical
  • Price range filters (e.g., $0 – $1,000)
  • Distance radius selection (e.g., 0 – 100 miles)
  • Weight or size ranges
Visual Feedback Matters
  • Settings where seeing the relative position provides context
  • Adjustments that benefit from immediate visual representation of the value
Mobile-optimized Input
  • Touch-friendly alternative to typing numeric values
  • Easier thumb dragging than keyboard input on mobile devices
Approximate Adjustments
  • When users are exploring or browsing rather than entering exact values
  • Filtering search results by flexible ranges
Precise Value Entry
  • Exact dollar amounts or financial inputs (use number input field instead)
  • Specific dates or times (use date/time picker)
  • Values requiring decimal precision
Binary or Few Options
  • On/off states (use switch/toggle)
  • 3–5 discrete choices (use radio buttons or segmented control)
  • Specific preset values (use dropdown or button group)
Critical or Legal Values
  • Financial transactions requiring exact amounts
  • Legal or contractual numbers
  • Any value where mistakes have serious consequences

Messaging

Group Label

Use clear, descriptive labels that tell users what the slider controls:

  • Keep labels concise and action-oriented.
  • Include required indicator (*) when the filter or input is mandatory.
  • Use title case for labels.
  • Keep labels under 50 characters for readability.

Example labels:

  • "Price Range"
  • "Delivery Distance"
  • "Shipment Weight Limit"

Helper Text

Provide additional context to guide users:

  • Explain what the range represents or what units are being used.
  • Clarify any constraints or limitations (e.g., "Based on your location").
  • Describe what happens when the value is adjusted.
  • Keep helper text brief (1–2 lines maximum).

Example helper text:

  • "Adjust to filter shipments within your preferred price range"
  • "Maximum weight capacity for this shipment type"
  • "Shows carriers available within this distance from your location"

Best Practices

✅ Do:

  • Always label sliders — unlabeled sliders create confusion.
  • Use helper text when the range or units might be unclear.
  • Match label terminology to what users see elsewhere in the app.
  • Test labels with actual users to ensure clarity.

❌ Don't:

  • Use vague labels like "Slider 1" or "Adjust value".
  • Write lengthy helper text that requires scrolling.
  • Include redundant information between label and helper text.
  • Use technical jargon unless the audience expects it.

Accessibility

  • ARIA role: Implement as role="slider" with aria-valuenow, aria-valuemin, and aria-valuemax so screen readers announce the current position.
  • Label association: Always associate a visible label using aria-labelledby or aria-label so screen reader users understand what is being controlled.
  • Keyboard support: Tab moves focus to the thumb. Arrow keys (← / →) adjust the value by one step. Home / End jump to min / max.
  • Value announcement: Update aria-valuenow as the value changes so screen reader users hear the new value without delay.
  • Focus visibility: The teal focus ring must be clearly visible. Do not suppress the focus indicator for keyboard users.
  • Contrast: The filled track and value badge meet WCAG AA contrast requirements against both light and dark surface backgrounds.
