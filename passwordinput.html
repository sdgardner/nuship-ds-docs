Password Input

A password input is a sub-variant of the text input used for entering passwords or other sensitive data. The value is masked by default (e.g. dots or bullets). It includes an optional show/hide control (trailing icon) so users can reveal the value to check what they typed. It uses the same structure as the base input (label, field, optional placeholder and helper text) and the same tokens, with the addition of the interactive trailing icon for visibility. Use it whenever the user needs to enter a password, PIN, or similar secret value.

Usage Guidelines

When to use
  • Use for passwords, PINs, or other sensitive single-line input that should be hidden by default
  • Provide a show/hide control (trailing icon) so users can reveal the value to check what they typed
  • Use appropriate autocomplete so password managers work (e.g. current-password, new-password)
Show/hide and icon
  • Put the visibility icon at the end of the field (trailing icon). Use one state for masked (show) and one for revealed (hide)
  • Make the control keyboard usable and give it an accessible name (e.g. "Show password" / "Hide password")
Helper text and errors
  • Use helper text when you have password rules (e.g. "Must be at least 8 characters")
  • Use error messages that explain what's wrong (e.g. "Password must be at least 8 characters")
  • Use placeholder examples like "Enter password" or "Create a password" (optional)
Behavior and security
  • Keep the value masked by default and use type="password" (or equivalent)
  • Allow paste so password managers and users can paste passwords
  • Use the same states as the base input (default, focus, error, disabled)
When to use
  • Don't use for non-sensitive fields (e.g. username, email); use a standard text input
  • Don't reveal the value by default; keep it masked unless the user chooses to show it
  • Don't remove the option to show/hide the password
Show/hide and icon
  • Don't rely on the icon alone; ensure it has an accessible label
Helper text and errors
  • Don't put the actual password in placeholder or helper text
  • Don't use vague errors like "Invalid password"
Behavior and security
  • Don't block paste to "improve security"; it hurts accessibility and password managers
  • Don't reveal the password by default; only when the user triggers show

Accessibility

  • Keyboard: Tab moves focus to the field. Tab again moves focus to the visibility toggle button. Enter or Space on the toggle reveals or conceals the password.
  • Screen readers: Use <input type="password"> for the field. Label the toggle button with aria-label="Show password" or "Hide password" and update it dynamically when toggled. Announce the state change with aria-live="polite".
  • Visual: Never remove the visibility toggle — it is a critical affordance for users with motor or cognitive impairments. Focus ring must be clearly visible (2px blue border + glow). Error state must not rely on colour alone — pair red border with the error icon and message text.
  • Touch targets: The visibility toggle icon area must be at least 44×44px on touch devices. Ensure adequate padding around the icon to prevent mis-taps with the input field.
